# CLAUDE.md — инструкции для Claude Code

Проект **my_map-poi**: статическая карта значимых достопримечательностей на Wikidata +
Wikipedia + Commons, скоринг вместо ручной курации. Полное ТЗ — в `SPEC.md`. Этот файл — про
то, КАК вести работу в репозитории.

## Контекст и неизменяемые рамки

- Источник объектов — **только Wikidata SPARQL по сетке под-bbox'ов, с фильтром по типу
  (`wdt:P31`, whitelist/blacklist из SPEC §1a)**. Никакого ручного списка мест, никакого
  полного дампа OSM tourism=*/historic=* без фильтрации по значимости, и никакой выборки
  «координаты + sitelink» без type-фильтра — иначе в карту попадут города/дороги/реки как
  «достопримечательности» (это уже обсуждали и зафиксировали как баг раннего дизайна).
- **Q-id для whitelist/blacklist не брать по памяти/из training data.** Проверять в Wikidata
  Query Service на реальном запросе (шаг 2 ниже) и фиксировать в `config/poi-types.json` с
  комментарием, что означает каждый код. Неверный Q-id молча даёт пустой/неполный результат —
  это не то место, где стоит гадать.
- Пилотный регион — **коридор Больцано–Верона** (bbox из `config/regions.json`, ~190×180 км)
  — заметно больше одного города, поэтому fetch **обязан** быть сеткой запросов, а не одним
  SPARQL-вызовом на весь bbox (риск таймаута/обрезанного результата на плотных городских
  клетках вроде Вероны). См. SPEC §2, §8.
- Обогащение (score, фото, summary, атрибуция) — **на этапе сборки данных**, не в рантайме
  браузера. Клиент не должен делать сетевые вызовы к Wikidata/Wikipedia/Commons API — все
  нужные поля уже лежат в properties тайла.
- Никакого бэкенда/БД/сервера — сайт полностью статический, как `my_map-toll`. Basemap —
  OpenFreeMap, оверлей — наши PMTiles через GitHub Release.
- Commons-фото **обязательно** с атрибуцией (автор + лицензия + ссылка), взятой из
  `imageinfo|extmetadata`. Если атрибуцию не удалось получить — фото не публикуется, точка
  остаётся без фото-блока. Не пропускать эту проверку ради красивой картинки.
- Flickr, несколько регионов одновременно, текстовый поиск, интеграция с routing-проектом —
  сознательно вне MVP (см. SPEC §7). Не добавлять по инициативе, даже если «легко было бы
  заодно».

## Порядок работы (вертикальными срезами)

1. **Скоринг как чистая функция.** `src/score.mjs` + юнит-тесты на выдуманных входных данных
   (sitelinks/pageviews/has_image → score). Это ядро, от которого зависит вся плотность
   точек среди уже отфильтрованных по типу объектов — должно быть проверяемо без единого
   сетевого вызова.
2. **Fetch на одной клетке + верификация P31-фильтра.** Сначала в Wikidata Query Service
   (query.wikidata.org) вручную/скриптом подобрать и проверить реальные Q-id для whitelist
   landmark-категорий и blacklist (населённые пункты, административные единицы и т.д.) из
   SPEC §1a — зафиксировать в `config/poi-types.json`. Затем реализовать SPARQL-запрос по
   одному под-bbox'у (например, маленькая клетка вокруг Тренто) с этим фильтром → сырой JSON
   на диск. Проверить на реальном ответе, что нужные поля (qid, coords, sitelinks, P18, P373)
   приходят и что в выдаче нет городов/дорог/рек — если есть, значит blacklist неполный,
   доработать раньше, чем нарезать сетку на весь регион. Только после этого — нарезка всего
   регионного bbox на клетки и прогон по всем с паузами между запросами + дедупликация по
   qid на границах клеток.
3. **Enrich:** pageviews + summary + commons-атрибуция по списку qid из шага 2. Кешировать
   промежуточный JSON на диск между шагами — на регион такого размера это не опция, а
   необходимость: переотправлять тысячи SPARQL/REST запросов при каждой правке скоринга
   недопустимо.
4. **Normalize + score:** собрать GeoJSON FeatureCollection, применить `score()` из шага 1.
5. **Tile + publish:** tippecanoe → PMTiles → GitHub Release, по образцу
   `my_map-toll/scripts/04-tile.sh` и `05-publish.sh` (те же инструменты, адаптировать пути).
6. **Web:** MapLibre + PMTiles + OpenFreeMap, попап по SPEC §5, zoom-зависимый score-фильтр.
   Начать с фиксированного порога, потом заменить на interpolate по zoom.
7. **Полировка:** пустые/частичные данные (нет фото, нет summary на нужном языке — в этом
   регионе актуален немецкий/итальянский на границе Южного Тироля), поведение при
   score-фильтре на границе zoom, `docs/DATA-LICENSES.md`.

Каждый шаг должен давать что-то проверяемое (лог/файл/скриншот карты), не «весь пайплайн
сразу, потом фронтенд». Особенно шаг 2 — сначала P31-фильтр проверен вручную на одной
клетке, потом сетка на весь регион, не наоборот.

## Конвенции

- Промежуточные данные каждого шага пайплайна — в `data/` (добавить в `.gitignore`, кроме
  небольшого примера-сэмпла для тестов), чтобы не пересчитывать всё с нуля при отладке
  одного шага.
- Магические числа скоринга (веса `w1/w2/w3`, cap на объекты, пороги zoom, размер клетки
  сетки для fetch) — в одном конфиг-месте, не разбросаны по скриптам. Их придётся крутить по
  реальным данным пилота.
- SPARQL-запросы — держать текстом в отдельных `.rq`-файлах или явных константах, не
  собирать строкой на лету из нескольких кусков — так проще отлаживать в Wikidata Query
  Service напрямую (query.wikidata.org/#вставить-и-проверить).
- Атрибуция фото — обязательное поле в normalized GeoJSON, не опциональное сбоку.
- `config/poi-types.json` (whitelist/blacklist Q-id) — с комментарием на каждый код, что это
  за категория; при добавлении новых категорий в процессе полировки — тоже проверять в WDQS,
  не угадывать.

## Чего НЕ делать без явного запроса

- Не добавлять Flickr или другие источники фото сверх Wikidata/Commons.
- Не публиковать фото без атрибуции автора/лицензии.
- Не делать сетевые вызовы к внешним API из браузерного кода — только на этапе сборки.
- Не делать один SPARQL-запрос на весь регион вместо сетки под-bbox'ов — на плотных клетках
  (Верона, Виченца) это риск таймаута/обрезанного результата.
- Не делать выборку объектов только по «координаты + sitelink» без P31-фильтра — так в карту
  попадут города, дороги, реки как «достопримечательности».
- Не вписывать Q-id классов Wikidata по памяти без проверки в WDQS.
- Не расширять на несколько регионов одновременно без запроса — конфиг это позволяет, но
  MVP — один регион (Больцано–Верона).
- Не заводить бэкенд/API-прокси — если кажется, что без него не обойтись, значит что-то
  спроектировано не так, стоит вернуться к SPEC.

## Сборка / запуск

Ожидаемо аналогично `my_map-toll`: `npm install`, `npm run data:build` (fetch → enrich →
score → normalize → tile), `npm run data:publish` (gh release upload), `npm run dev` для
локального фронтенда. Конкретные npm-скрипты завести по мере реализации шагов из раздела
выше — не обязательно копировать имена один в один с `my_map-toll`, если структура проекта
проще.
