# my_map-poi — техническое задание

Статическая веб-карта значимых достопримечательностей региона. Источник —
Wikidata + Wikipedia + Wikimedia Commons (никакого ручного кураторства). Каждая точка на
карте — с миниатюрой фото, коротким описанием и ссылкой на Google Maps. Плотность точек
регулируется скорингом значимости, а не полным списком OSM tourism=*/historic=*.

Архитектурно и по духу — сестринский проект `my_map-toll`: статический сайт, PMTiles,
OpenFreeMap basemap, публикация через GitHub Pages/Releases, без бэкенда. Но это
**отдельный репозиторий**, не пакет внутри `my_map-toll` — пайплайны обогащения
принципиально разные (сетевой fetch+score против чистой OSM-tag интерпретации), и мешать их
в одном месте не стоит.

---

## 1. Зафиксированные решения

| Тема | Решение |
|---|---|
| Источник объектов | **Wikidata SPARQL** по сетке под-bbox'ов — объекты с координатами (`wdt:P625`), у которых есть хотя бы один sitelink, **и** тип (`wdt:P31`) входит в whitelist landmark-категорий (см. §1a). Не зависит от OSM-тегов вообще. |
| Скоринг значимости | Комбинация sitelinks + pageviews + наличие фото. Формула — см. §3. Управляет плотностью точек на карте среди уже отфильтрованных по типу объектов. |
| Фото (1–2 шт.) | **Wikidata P18** (image) для объекта; если пусто — 1–2 файла из Commons-категории объекта (`P373`) через Commons API. Хотлинк на `upload.wikimedia.org`, с атрибуцией автора/лицензии из `imageinfo|extmetadata`. |
| Краткое описание | **Wikipedia REST Summary API** (`/page/summary/{title}`) — один абзац + ссылка на статью. Плюс короткий `description` из Wikidata как подпись под названием. |
| Ссылка на Google Maps | Генерируется на лету из координат: `https://www.google.com/maps/search/?api=1&query={lat},{lon}`. Ключ не нужен. |
| Обогащение — когда | **На этапе сборки данных**, не в рантайме браузера. Готовые поля (score, image_url, image_attribution, summary, wiki_url) кладутся в properties фичи GeoJSON → тайлятся. Клиент не делает сетевых вызовов к Wikidata/Wikipedia/Commons. |
| Basemap | OpenFreeMap (как в `my_map-toll`), keyless. |
| Оверлей | Наши PMTiles с точками POI, публикуются через GitHub Release. |
| Регион MVP | **Коридор Больцано–Верона** (Южный Тироль, Трентино, часть Венето и краешек Ломбардии) — прибл. bbox `9.75, 45.15, 12.35, 46.75` (west, south, east, north), ~190×180 км, ~34 000 км². Заметно больше «одного города» — включает несколько плотных по достопримечательностям центров (Больцано, Тренто, Верона, Виченца), поэтому fetch делается сеткой под-bbox'ов (см. §2), а не одним SPARQL-запросом. Полная Европа — не MVP, см. §9. |
| Flickr-слой | **Не в MVP.** Wikidata/Commons достаточно для пилота и не требует API-ключа на клиенте. Точка расширения оставлена (см. §7), но не реализуется сейчас. |
| Поисковый движок объектов | Только SPARQL-запрос по bbox + type-фильтр, без ручного списка мест. |

## 1a. Фильтр по типу объекта (P31)

Без фильтра по типу «координаты + sitelink» пропускает не только достопримечательности, но
и населённые пункты, административные единицы, дороги, реки и т.п. — у города Верона как
объекта Wikidata sitelinks/pageviews будут выше, чем у любой конкретной достопримечательности
внутри него, и скоринг (§3) сам по себе это не отсеет.

Решение — **whitelist категорий через `wdt:P31`** (instance of), плюс **blacklist на
исключение**, даже если объект случайно попал в оба списка:

**Whitelist (примерные категории, не точные Q-id):** замок, дворец, монастырь, церковь /
базилика / храм, музей, памятник, скульптура/произведение искусства, археологический
памятник, смотровая площадка, водопад, пещера, крепость/укрепление, башня, знаковый мост,
фонтан, природный памятник, гора/перевал (для сценического контекста — актуально в этом
регионе).

**Blacklist (исключить, даже с sitelink):** населённый пункт (`human settlement` и
подклассы — город/посёлок/деревня), административно-территориальная единица, дорога/шоссе,
водоток (кроме отдельно поименованных водопадов), железнодорожная линия (не станция).

**Важно:** точные Q-id для whitelist/blacklist в этот документ намеренно не вписаны —
их нужно проверить в Wikidata Query Service на реальных данных (шаг 2 в `CLAUDE.md` и так
начинается с одной тестовой клетки — там же свериться с правильными кодами классов, а не
брать их по памяти). Ошибочный Q-id в SPARQL молча даст пустой или неверный результат.

## 2. Поток данных

```
config/regions.json: один регион → нарезается на сетку под-bbox'ов (напр. 0.3°×0.3°)
  ▼
Wikidata SPARQL endpoint (query.wikidata.org), по одному под-bbox'у за раз, с паузой между запросами
  │  под-bbox + P31 whitelist/blacklist (§1a) → { qid, lat, lon, sitelinks_count,
  │  image (P18), commons_category (P373), label, description }
  │  результаты по всем под-bbox'ам дедуплицируются по qid (объекты на границах клеток)
  ▼
Wikipedia REST API (pageviews + summary по sitelink на нужном языке)
  │  qid → { pageviews_90d, summary_text, wiki_url }
  ▼
Commons API (imageinfo)
  │  image filename / category → { image_url, thumb_url, author, license, license_url }
  ▼
Скоринг (чистая функция, юнит-тестируемая)
  │  → score = w1*log(sitelinks+1) + w2*log(pageviews+1) + w3*has_image
  ▼
Normalize → GeoJSON features
  │  properties: { qid, name, description, score, image_url, image_attribution,
  │                summary, wiki_url, maps_url (генерится на клиенте из geometry) }
  ▼
tippecanoe → PMTiles
  ▼
GitHub Release (poi-overlay.pmtiles)
  ▼
Web (MapLibre + PMTiles + OpenFreeMap): маркер размер/цвет по score,
  zoom-зависимый порог score, попап с фото+описанием+ссылками
```

Никакого сервера. Обновление данных — ручная команда или редкий scheduled workflow (раз в
месяц-два — sitelinks и картинки почти не меняются; pageviews можно освежать чаще, но это не
критично для MVP). Из-за размера пилотного региона fetch+enrich могут занять часы, а не
минуты — промежуточный кэш на диске (см. `CLAUDE.md`) обязателен, не опционален.

## 3. Скоринг

```
score = w1 * log(sitelinks_count + 1)
      + w2 * log(pageviews_90d + 1)
      + w3 * (1 if has_image else 0)
```

Применяется **после** type-фильтра §1a, не вместо него — скоринг регулирует плотность среди
уже отобранных landmark-категорий, а не отделяет достопримечательности от городов/дорог/рек.

Стартовые веса: `w1 = 1.0, w2 = 1.0, w3 = 0.5` — не константа истины, крутить по результатам
пилота на реальном регионе. Объекты без единого sitelink в фильтр не попадают вообще (не
Wikidata-значимые = не «столбы и памятники», а действительно неизвестные точки, которых в
принципе не должно быть на этой карте).

Zoom-зависимая плотность на клиенте: чем дальше zoom, тем выше порог `score` для показа
точки (аналог MapLibre `filter` с interpolate по zoom). Точных порогов заранее не знаем —
подбираются на глаз по пилотному региону. На таком разнородном регионе (горы + несколько
городов) стоит проверить, не перекашивает ли скоринг в пользу городских объектов в ущерб
природным точкам (перевалы, смотровые) — у последних sitelinks/pageviews обычно ниже при
не меньшей реальной значимости.

## 4. Структура репозитория

Проще, чем у `my_map-toll` — нет отдельного domain-interpreter, поэтому монорепо из 5
пакетов не нужно. Предлагаемая раскладка:

```
scripts/
  01-fetch-wikidata.mjs      # регион → сетка под-bbox'ов → SPARQL (bbox+P31-фильтр) → raw JSON
  02-enrich.mjs              # + pageviews, summary, commons photo/attribution
  03-score.mjs               # чистая функция score(), юнит-тесты рядом
  04-normalize.mjs           # → GeoJSON FeatureCollection
  05-tile.sh                 # tippecanoe → PMTiles
  06-publish.sh              # gh release upload
src/
  score.mjs                  # экспортируемая чистая функция + score.test.mjs
web/
  index.html, src/, vite.config.ts   # MapLibre + PMTiles + OpenFreeMap, попап
.github/workflows/
  data.yml                   # ручной запуск/редкий cron — пересборка PMTiles
  pages.yml                  # деплой web на каждый push в main
docs/
  ARCHITECTURE.md
  DATA-LICENSES.md           # что откуда, какая атрибуция обязательна
config/
  regions.json                # регион(ы): общий bbox + размер клетки сетки
  poi-types.json              # whitelist/blacklist Q-id из §1a (заполняется при реализации)
```

## 5. Попап объекта (фронтенд)

Поля, в порядке отображения:

1. Фото (если есть `image_url`) + подпись атрибуции (`image_attribution`, мелким шрифтом,
   ссылка на файл на Commons).
2. Название + `description` из Wikidata (короткая подпись-категория).
3. `summary` — абзац из Wikipedia.
4. Ссылка «Подробнее в Wikipedia» → `wiki_url`.
5. Ссылка «Открыть в Google Maps» → генерируется из координат фичи.

Если `image_url` пуст — просто нет фото-блока, попап не ломается.

## 6. Лицензии и атрибуция

| Источник | Лицензия | Что делать |
|---|---|---|
| Wikidata | CC0 | Атрибуция не обязательна, но ссылка на объект — хорошая практика. |
| Wikipedia summary | CC BY-SA | Обязательна ссылка на статью (уже есть в попапе, п.4 §5). Текст не переписывать/не обрезать до неузнаваемости — берём summary как есть от API. |
| Commons images | Разная по файлу (CC-BY, CC-BY-SA, PD) | **Обязательно** тянуть `imageinfo|extmetadata` (автор, лицензия, ссылка на файл) и показывать в попапе. Без этого поля фото не публикуем. |

Зафиксировать в `docs/DATA-LICENSES.md` при реализации — это единственное место в проекте,
где легко случайно накосячить с чужими правами.

## 7. Явно НЕ в MVP

- Flickr как источник фото/плотности — точка расширения, не реализуется сейчас.
- Несколько регионов одновременно — конфиг это позволяет, но пилот — один регион (Больцано–Верона).
- Текстовый поиск по названию (Nominatim) — как и в `my_map-toll`, не в этой версии.
- Интеграция с scenic-detour/маршрутным проектом — POI отсюда потенциально станут входом
  для того проекта, но это отдельная задача, не эта.
- Автоматический еженедельный/ежедневный cron на обновление pageviews — старт с ручного
  запуска workflow, как `data.yml` в `my_map-toll`.
- Вся Европа сразу — см. §9, требует другого fetch-механизма (dump вместо live SPARQL).

## 8. Открытые вопросы

- Точный список Q-id для whitelist/blacklist из §1a — уточняется в Wikidata Query Service
  на первой тестовой клетке (шаг 2 в `CLAUDE.md`), не берётся по памяти. Возможно, список
  придётся дополнять по мере обнаружения пропущенных категорий на реальных данных региона.
- Точные веса скоринга и пороги zoom-фильтрации — калибруются на пилотном регионе, не
  угадываются заранее.
- Язык summary/wiki-статьи при мультиязычных sitelinks — в этом регионе явно актуально
  (немецкий/итальянский на границе Южного Тироля): дефолт — итальянский, для объектов с
  преимущественно немецким sitelink — рассмотреть fallback на него, иначе английский.
  Уточнить при первом реальном прогоне.
- Размер клетки сетки под-bbox'ов для fetch — компромисс между числом запросов и риском
  упереться в лимит результатов/таймаут на плотных клетках (города). Стартовое значение
  0.3°×0.3° — подбирается по факту первого прогона.
- Верхний лимит объектов на клетку (SPARQL может вернуть тысячи в плотной городской клетке)
  — нужен ли hard cap или достаточно порога по score на этапе normalize.

## 9. Масштабирование до Европы (roadmap, не MVP)

Соблазн сразу сделать «как в `my_map-toll`, вся Европа» — обманчив: там массовая обработка
идёт по **локальному файлу** (Geofabrik-экстракт), а здесь — по **живым публичным API**
(Wikidata SPARQL endpoint, Wikipedia summary/pageviews, Commons imageinfo), у которых нет
bulk-режима и есть неформальные лимиты вежливого использования. Пилотный регион уже
вынуждает делать fetch сеткой под-bbox'ов (§2) — на всей Европе тот же принцип, но клеток
уже не десятки, а тысячи, и это совсем другая по масштабу задача.

Путь масштабирования, когда до него дойдёт:

- **Fetch** — заменить live SPARQL по сетке под-bbox'ов на локальную фильтрацию
  **Wikidata JSON dump** (полный дамп, скачивается и парсится так же, как Geofabrik-экстракт
  в `my_map-toll`) — по координатам + наличию sitelinks + P31-фильтру из §1a. Это снимает
  ограничение публичного endpoint полностью, а сеточная логика из MVP переиспользуется как
  есть для разбивки результата на тайлы для normalize/tile-builder.
- **Enrich** (Wikipedia summary/pageviews, Commons imageinfo) остаётся per-item запросами —
  официального bulk-эндпоинта у них нет. На масштабе Европы это решается регион-за-регионом
  с резюмируемым диск-кэшем (уже заложено в `CLAUDE.md`, шаг 3 порядка работы), а не одним
  прогоном на весь континент. Стоит рассмотреть MediaWiki Action API
  (`prop=extracts|pageimages`, до 50 titles за запрос) вместо REST Summary API для
  сокращения числа запросов, если/когда до этого дойдёт.
- Интерфейс "bbox → список qid с нужными полями" в `01-fetch-wikidata.mjs` должен остаться
  таким, чтобы реализацию позже можно было подменить на dump-based без изменений в
  enrich/score/normalize.

Не реализуется сейчас — фиксируется здесь, чтобы порядок работы в `CLAUDE.md` не пришлось
переписывать с нуля при переходе на континентальный масштаб.
